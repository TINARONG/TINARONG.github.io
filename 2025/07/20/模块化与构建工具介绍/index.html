多场景构建工具介绍与选型

1.模块化介绍

1.1 模块化目的

  1. 将一个复杂的程序，依据一定的规则封装成几个块（文件），并组合在一起
  2. 块的内部数据和实现是私有的，只向外暴露一些接口（方法）与外部进行通信；

2 模块化规范

2.1 模块化发展

2.1.1 全局function模式

编码：将不同模块封装成不同的全局函数
问题：污染全局命名空间，可能造成命名冲突或数据不安全，而且模块之间看不出关系

2.1.2 namespace命名空间模式

编码：简单对象封装，减少了全局变量
问题：数据不安全

2.1.3 IFFE模式

编码： 匿名函数闭包，通过向window添加属性暴露方法；数据私有，解决外部修改问题
问题： 无法解决模块之间的以来问题

2.1.4 模块化发展时间线

commonjs => amd => cmd => es6 => umd

commonjs

  • 动态导入
  • node.js天然支持
  • 无法做tree-shaking(由于动态导入，不好分析，并且主要用于服务端、不需要考虑代码大小、没有必要做)

amd

  • 使用requirejs

cmd

整合和commonjs和amd规范的特点，并在sea.js实现了落地

es modlue √

  • 关于打包：
    虽然目前浏览器已经支持esm,但webpack打包还是会转umd/iife
    vite的打包是做少量化编译，会保留部分内容
  • 针对nodejs支持，2个方案
    1. 添加package.json type: “module”;
    2. 文件名为.mjs

umd(通用化模块定义)

6 选型tips

  1. 任意方向需要有至少3个以上的选型方案比较选择；
  2. 选型关注点：
    • 项目的规模和复杂度
    • 团队对工具的熟悉程度
    • 工具的使用和开发体验
    • 项目是否需要高度自定义的打包流程
    • 项目是否有特殊的任务自动化需求
  3. 一般项目选择
    业务项目
    webpack: vite生态库不足以支持的（例如微前端、低代码等），老项目;
    vite: 新兴项目、快速构建；
    库类（组件库、工具库等）
    rollup: 一般vue生态、js编程推荐；
    tsup: react生态，ts支持项目推荐；
